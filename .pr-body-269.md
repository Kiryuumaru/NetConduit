# fix(mux): route channel to accept queue when pending TCS already cancelled (fixes #269)

## Problem

`ChannelRegistry.AcceptChannelAsync(channelId, ct)` and `ChannelRegistry.EnqueueForAccept(channel)` race on the same `ConcurrentDictionary<string, TaskCompletionSource>` and the same TCS instance via two read-modify-write sequences with no coordination between them:

- `EnqueueForAccept`: `TryRemove` then `TrySetResult`.
- Cancellation callback: `TryRemove` then `TrySetCanceled`.

When the cancellation callback runs between `EnqueueForAccept`'s `TryRemove` and its `TrySetResult`, the cancellation callback's own `TryRemove` returns `false` (the entry was already removed), but it unconditionally calls `tcs.TrySetCanceled(ct)`. The TCS flips to `Canceled`. `EnqueueForAccept` then resumes, calls `tcs.TrySetResult(channel)`, gets back `false`, and **silently drops the channel**:

- The `ReadChannel` is fully alive in `_readChannels` (registered by `DispatchToChannel(Init)` before `EnqueueForAccept` ran).
- It is **not** delivered to the awaiter (they get `OperationCanceledException`).
- It is **not** routed to `_acceptQueue` for `AcceptChannelsAsync(null)`.
- It is **not** routed to a matching prefix subscription.
- The peer believes the channel is open and keeps writing into the orphan slab where no consumer can read it.

## Fix

Two coordinated changes in `src/NetConduit/Internal/ChannelRegistry.cs`:

### 1. `EnqueueForAccept` — fall through when `TrySetResult` fails

```csharp
if (_pendingAccepts.TryRemove(channel.ChannelId, out var tcs)
    && tcs.TrySetResult(channel))
{
    return;
}
// fall through to prefix-subscription / _acceptQueue routing
```

If the awaiter already cancelled (or otherwise completed) the TCS we just removed, route the channel through the catch-all paths so the host can still observe it via `AcceptChannelsAsync` or a prefix subscription instead of leaking it.

### 2. Cancellation callback — only cancel the TCS we still own

```csharp
await using var registration = ct.Register(() =>
{
    if (_pendingAccepts.TryRemove(channelId, out _))
        tcs.TrySetCanceled(ct);
});
```

If `TryRemove` returns `false`, the entry was already removed by `EnqueueForAccept` and that path will deliver the result. Calling `TrySetCanceled` here would race with `TrySetResult` and resurrect the original bug, so we skip it.

The two fixes are belt-and-braces: each independently closes one direction of the race; together they guarantee both correct delivery (channel is never dropped) and correct cancellation observation (caller sees `OperationCanceledException` only when `EnqueueForAccept` actually didn't claim the TCS).

  - `EnqueueForAccept_TcsAlreadyCancelled_RoutesChannelToAcceptQueue`: pre-stuffs `_pendingAccepts` with an already-cancelled TCS (matching the post-race state), calls `EnqueueForAccept`, asserts the channel arrives via the catch-all `AcceptChannelsAsync(null)` stream.
