# fix(transit): make `Ready` event latching so subscribers added after channels are ready still observe it (fixes #266)

## Problem

`DuplexStreamTransit`, `StreamTransit`, `MessageTransit<,>`, and `DeltaMessageTransit<>` all wire up their `Ready` event from underlying channel events in their constructors. The underlying channel `Ready` event is **single-shot**: `WriteChannel.MarkOpen` / `ReadChannel.MarkOpen` raise it exactly once gated by an `_isReady` flag and never fire again.

The natural calling pattern is to wait for channel readiness *before* wrapping in a transit:

```csharp
var write = mux.OpenChannel("id");
var read  = await peer.AcceptChannelAsync("id", ct);
await Task.WhenAll(write.WaitForReadyAsync(), read.WaitForReadyAsync());
var transit = new DuplexStreamTransit(write, read); // <-- channels already Ready here
transit.Ready += (_, _) => Console.WriteLine("ready"); // <-- never fires
```

Inside the constructor, `SubscribeToChannelEvents()` attaches a handler to each channel's `Ready` event, but those events have already fired. The transit's `Ready` event therefore never fires for the lifetime of the object. `IsReady` (a property that reads channel state on demand) still works — only the *event* is broken.

## Fix

Convert the auto-implemented `Ready` event into a latching custom event in all four transits:

- `_readyFired` (existing) tracks whether the transit has reached the Ready state
- The `add` accessor invokes the new handler immediately if `_readyFired` is already `true`
- `OnChannelReady` snapshots and invokes the current handler list under the lock that gates `_readyFired`, then sets `_readyFired = true` so subsequent `add` calls fire-on-attach

This makes the order of "channels become ready" vs "subscribe to `Ready`" irrelevant — late subscribers always observe the event exactly once, just like `IsReady` already worked.
